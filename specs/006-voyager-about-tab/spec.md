# 功能规格说明：Voyager ER 图 —— About Tab（docstring + Mermaid 渲染）& 侧边栏宽度放宽

**功能分支**：`006-voyager-about-tab`

**创建日期**：2026-07-01

**状态**：草稿

**输入**：用户描述："我需要在 voyager 的 er diagram 中，双击 entity 之后提供的信息做一个增强，具体做法是添加一个单独的 about tab，里面显示的内容是 docstring，但是需要支持 markdown 渲染，注意要支持 mermaid 的渲染，因为里面会添加一些状态转移之类的可视化信息。另外，我需要放宽侧边栏的拖拽区间，现在允许值太小了，我要允许它往左最多占用 2/3 的宽度。"

## 澄清记录

### Session 2026-07-01

- Q: 新增的 About tab 应该出现在侧边栏 tab 栏的哪个位置？首次打开侧边栏时哪个 tab 默认激活？ → A: 最左（出现在 Fields 之前），且**不改变**默认激活策略——Fields 仍是侧边栏首次打开时的默认激活 tab。语义上 About 是总览/入口放最前，但不打扰现有用户对"打开侧边栏看 Fields"的肌肉记忆。
- Q: About tab 中渲染的 docstring 从哪里取？ → A: 取自 schema 模型类的 Python `__doc__`（类级文档字符串）。这意味着后端需要在 schema 元数据中加入 docstring 字段，或新增一个返回该 docstring 的端点；不与字段级 `description` 混用，也不做 "__doc__ 空时回退到 description" 的合并。
- Q: docstring 中的 Markdown 链接点击后应该发生什么？ → A: 全部按普通 `<a>` 标签渲染，**不触发**任何对其他实体的跳转——外部 http/https 链接照常在新 tab 打开，其他链接渲染为可点击但不导航。与 Mermaid 只读、Related Entities 只读保持一致；实体内导航是后续 feature，不在本期范围。
- Q: Mermaid 语法错误导致渲染失败时，About tab 该如何展示该段？ → A: 错误提示 + **默认折叠**的原始 mermaid 源码。错误提示让用户知道图挂了；折叠源码让有需要的用户可以展开查看/复制到外部工具（如 mermaid.live）调试；默认折叠避免污染 docstring 阅读流。

## 用户场景与测试 *(必填)*

### 用户故事 1 —— 在 About tab 中以富文本形式阅读实体 docstring（优先级：P1）

开发者或数据探索者正在 Voyager 的 ER 图 tab 中浏览 schema。他们双击某个实体 `X`，侧边栏打开（包含 Fields / Source Code / Related Entities 三个 tab）。他们希望切换到一个新的 "About" tab，看到 `X` 这个实体类对应 docstring 的**人类可读渲染**——标题、段落、列表、表格、行内代码、代码块等都按 GitHub-Flavored Markdown 的方式排版——而不是一段枯燥的纯文本。这样他们能在不打开 Source Code、不读 Python 源码的前提下，快速理解这个实体"是什么、用来做什么、有哪些约束"。

**为什么是这个优先级**：这是本功能的核心价值——把已有的 docstring 从"埋在源码里、几乎没人读"变成"侧边栏里一眼可见的富文本说明"。即使不渲染 Mermaid、即使侧边栏宽度不变，仅 Markdown 渲染本身就能独立交付可感知的价值，因此是 P1。

**独立测试**：在 `demo/enterprise_voyager` 上打开 ER 图，双击一个 docstring 非空且含有标题/列表/代码块的实体 `X`，切换到 "About" tab；验证内容以排版良好的富文本呈现（标题字号有层级、列表有缩进、代码块有等宽字体），且与该实体类 `__doc__` 中的内容一致。

**验收场景**：

1. **给定** ER 图 tab 已加载某个 schema，**当** 用户双击实体 `X` 并切换到 "About" tab，**那么** tab 内展示 `X` 对应 docstring 的 Markdown 渲染结果，包含至少以下元素的合理排版：标题、段落、无序/有序列表、行内代码、围栏代码块、表格。
2. **给定** "About" tab 已为实体 `X` 打开，**当** 用户在画布上单击（侧边栏已打开时）或双击（侧边栏已关闭时）另一个实体 `Y`，**那么** About tab 立即刷新为 `Y` 的 docstring 渲染结果。
3. **给定** 实体 `X` 的 docstring 为空（或不存在），**当** 用户打开 "About" tab，**那么** tab 内展示一句明确、可读的空状态文案（如"该实体暂无 docstring"），让用户能区分"没有 docstring"和"加载中"或"出错"。
4. **给定** docstring 数据请求失败，**当** "About" tab 被激活，**那么** tab 内展示一个明确、可读的错误状态，与空状态相区分（不悄悄显示空白）。
5. **给定** "About" tab 渲染期间数据仍在加载，**当** 用户切换到该 tab，**那么** 出现可见的加载指示（如进度条或 spinner），避免误以为内容已加载完毕。
6. **给定** 用户在 About / Fields / Source Code / Related Entities 之间切换当前激活 tab，**当** 然后切换到另一个实体，**那么** 当前激活的 tab 保留（沿用 spec 005 FR-012 的策略），即如果 About 是激活态，切换实体后新实体的 About tab 直接呈现。
7. **给定** 实体 `X` 的 docstring 含 Markdown 链接（含指向其他实体的相对链接、外部 https 链接），**当** 用户在 About tab 内点击其中任意一个链接，**那么** 外部 https 链接照常在新标签页打开；其他链接**不**触发对其他实体的导航，主画布选中状态保持 `X` 不变，侧边栏仍停留在 `X` 的 About tab。

### 用户故事 2 —— 在 About tab 中查看 Mermaid 图表（如状态转移图）（优先级：P2）

在阅读 docstring 时，文档作者常常会把"实体生命周期"、"状态转移"、"与其他角色的交互流程"这类信息写成 Mermaid 图（`stateDiagram-v2`、`flowchart`、`sequenceDiagram` 等）。用户希望在 About tab 中，docstring 里所有 ```mermaid 围栏代码块都被**就地渲染成可视化图表**——而不是以原始 Mermaid 源码文本展示。这样他们能直接看到一张图，而不用心智地把源码翻译成结构。

**为什么是这个优先级**：Mermaid 渲染能显著提升含图 docstring 的可读性，但即使 Mermaid 不渲染（退化成代码块），Story 1 的纯 Markdown 价值也已交付，因此是 P2。

**独立测试**：人为构造一个实体 `X`，其 docstring 内含至少一个 ```mermaid 围栏代码块（例如 `stateDiagram-v2` 表达状态转移），打开 "About" tab；验证该代码块被渲染为一张可视化的状态图（节点、边、标签可见），而非原始文本；并验证同一个 docstring 中的非 mermaid 围栏代码块（如 ```python）仍按普通代码块渲染、不被误当作图表。

**验收场景**：

1. **给定** 实体 `X` 的 docstring 含有至少一段 ```mermaid 围栏代码块（合法语法），**当** "About" tab 被渲染，**那么** 该段被渲染为一张可视化 Mermaid 图（节点/边/标签清晰可辨），而不是原始源码。
2. **给定** docstring 同时含有 ```mermaid 与 ```python（或其他语言）围栏代码块，**当** "About" tab 被渲染，**那么** mermaid 块渲染为图表、其他代码块按等宽字体原样展示，二者不互相干扰。
3. **给定** docstring 中的 mermaid 源码存在语法错误，**当** "About" tab 试图渲染该图，**那么** 该段降级为"错误提示 + 默认折叠的原始源码"组合（错误信息明确、源码默认折叠、可手动展开复制），不出现整页崩溃，并保留 docstring 其余部分（含其他 mermaid 块）的正常渲染。
4. **给定** 多个 mermaid 围栏块出现在同一 docstring，**当** About tab 渲染，**那么** 每一块都被独立渲染为独立的图表，互不影响。
5. **给定** mermaid 图被成功渲染，**当** 用户切换实体后再切回 `X`，**那么** 图表能正确重新渲染（不出现残留、空白或重复渲染）。

### 用户故事 3 —— 拖宽侧边栏至最多占视窗 2/3 宽度（优先级：P2）

无论是阅读 About tab 中较宽的 Markdown 表格、Mermaid 图，还是查看 Source Code 中较长的 Python 行，用户都常常觉得侧边栏太窄、内容被频繁换行或横向滚动。当前侧边栏的拖拽区间上限定死了（在视窗足够大时仍偏窄）。用户希望能把侧边栏**往左拖到最多占视窗宽度的 2/3**，从而在阅读宽内容时拥有足够横向空间；同时保留下限，避免侧边栏过窄不可用。

**为什么是这个优先级**：放宽拖拽区间能立刻提升所有 tab（About / Fields / Source Code / Related Entities）的可读性，但本身不引入新功能，是体验增强，因此 P2。

**独立测试**：在足够宽的桌面上（例如 1920px 视窗），双击实体打开侧边栏，按住拖拽手柄向左拖；验证可以把侧边栏宽度拖到至少 1280px（≈ 1920 × 2/3）附近；再向右拖回；验证下限仍能拖到合理最小宽度（不挤崩 UI）。

**验收场景**：

1. **给定** 一个宽度为 `W` 像素的视窗，**当** 用户拖动侧边栏拖拽手柄向左到极限位置，**那么** 侧边栏宽度不超过 `floor(W × 2/3)`（允许 ±1px 取整误差）。
2. **给定** 视窗宽度为 1920px，**当** 用户拖到极限，**那么** 侧边栏可达到约 1280px 宽（不再被旧有的固定上限卡住）。
3. **给定** 用户拖拽侧边栏，**当** 向右拖到下限，**那么** 侧边栏宽度不低于一个合理的最小可用宽度（不破坏 Fields/Source Code 等已有 tab 的可用性）。
4. **给定** 侧边栏已被拖到当前视窗的 2/3 宽度，**当** 用户缩小浏览器视窗（例如从 1920px 缩到 1200px），**那么** 侧边栏宽度自动收缩到新的 2/3 上限（≈ 800px），不会出现"侧边栏比画布还宽"的破损布局。
5. **给定** 用户刷新页面或重新打开侧边栏，**当** 上一次拖拽设定了一个 ≥ 当前 2/3 上限的宽度（因视窗变小），**那么** 重新打开时侧边栏宽度被 clamp 到当前 2/3 上限，不会超出。
6. **给定** 侧边栏处于关闭状态或刚打开的初始宽度（约 300px），**当** 用户正常使用、不主动拖拽，**那么** 侧边栏宽度行为与之前一致（这一故事不改变默认宽度，只放宽上限）。

### 边界情况

- docstring 完全为空、或仅为空白字符：显示明确空状态文案（见 FR-006）。
- docstring 仅含 Mermaid、没有任何 Markdown 段落：About tab 仍渲染该 Mermaid 图，不应因缺少段落而崩溃。
- docstring 含有 Mermaid 之外的不受支持图表语言（如 `plantuml`、`dot`）：按普通代码块展示，不渲染。
- docstring 含有恶意或破坏性 HTML（`<script>`、`onerror=` 等）：渲染管线必须清洗/剥离危险内容，不允许执行任意脚本（见 FR-009）。
- docstring 含有外部图片：默认允许渲染（标准 Markdown 图片语法），但失败时降级为断图图标，不影响其余内容。
- docstring 极长（数千行）：About tab 内容区可纵向滚动，不撑破侧边栏；不做强制分页。
- 同时多个 Mermaid 图、且其中一个语法错误：错误的那个降级为错误提示，其余正常渲染（见 Story 2 验收场景 3）。
- 视窗非常窄（移动端 / 小窗）：2/3 上限仍可能过小，但下限保证最小可用宽度；超出当前上限时按上限 clamp。
- 视窗非常大（4K 屏）：2/3 上限可能让侧边栏显得过宽，但这是用户主动拖拽的结果；不主动施加新的更小上限。
- 用户拖到极限后切换实体：宽度保持，不被重置；切换实体只刷新内容，不动宽度。
- 用户拖到极限后刷新页面：宽度若仍在当前 2/3 上限内则保持；超出则 clamp 到当前上限。
- About tab 被激活时实体被删/重命名：展示明确错误状态，不要悄悄空白。
- 在 About tab 已打开时切换到另一个 tab 再切回：内容仍应保留（不重复发请求），渲染结果不丢失。

## 需求 *(必填)*

### 功能需求

#### About Tab（docstring + Markdown + Mermaid）

- **FR-001**：Voyager ER 图侧边栏（双击实体后打开、当前含 Fields / Source Code / Related Entities 三个 tab 的那个面板）必须额外提供一个名为 "About" 的 tab；该 tab 在 ER-diagram 模式下可见，且在 tab 栏中位于最左位置（出现在 Fields 之前）。侧边栏首次打开时的默认激活 tab 维持 Fields，不因新增 About 而改变。
- **FR-002**：About tab 的内容必须取自当前所选 schema 模型类的 Python `__doc__`（即类级文档字符串），且每次所选实体变化时刷新。**不与**字段级 `description` 字段混用，也不做"类 `__doc__` 空时回退到 description"的合并；字段级 `description` 仍按现有方式在 Fields tab 内呈现。
- **FR-003**：About tab 必须把 docstring 作为 GitHub-Flavored Markdown 渲染，至少正确呈现：标题（H1–H6）、段落、无序/有序列表、行内代码、围栏代码块（含语法高亮可选）、表格、链接、引用块、加粗/斜体、水平分割线。
- **FR-004**：About tab 必须把 docstring 内的所有 ```mermaid 围栏代码块就地渲染为可视化图表（覆盖至少 `stateDiagram-v2`、`flowchart`、`sequenceDiagram` 三种常见类型），而非展示原始源码。
- **FR-005**：当 docstring 为空、缺失、或仅含空白字符时，About tab 必须展示一句明确、可读的空状态文案（如"该实体暂无 docstring"），让用户能区分"没有 docstring"与"加载中"或"出错"。
- **FR-006**：当 docstring 数据无法获取时，About tab 必须展示一个明确、可读的错误状态，且与空状态相区分。
- **FR-007**：在 About tab 内容加载期间，必须展示可见的加载指示（如线性进度条或 spinner）。
- **FR-008**：About tab 的内容必须随所选实体变化而刷新；并且切换实体时，若 About 是当前激活 tab，新实体打开后 About 仍为激活 tab（沿用 spec 005 FR-012 的"切换实体保留激活 tab"策略，对四个 tab 一视同仁）。
- **FR-009**：About tab 的 Markdown/HTML 渲染管线必须对危险内容做清洗（剥离 `<script>`、事件处理器属性如 `onerror`、`javascript:` 链接等），防止 docstring 中的任意文本被执行为脚本。
- **FR-017**：docstring 中的所有 Markdown 链接必须按普通 `<a>` 标签渲染：外部 `http`/`https` 链接照常在新 tab 打开；其他链接（含看上去像实体引用的相对链接）渲染为可点击样式但**不**触发对其他实体的导航、不改变主画布选中状态。About tab 与 Mermaid 图、Related Entities 子图一样只读。

#### Mermaid 渲染

- **FR-010**：当 docstring 中某段 mermaid 源码存在语法错误导致渲染失败时，该段必须降级展示为"**错误提示 + 默认折叠的原始 mermaid 源码**"组合：错误提示明确告知该图渲染失败；折叠区在用户主动展开后展示原始源码以便复制到外部工具调试；默认折叠状态不污染 docstring 的整体阅读流。该降级不得影响 docstring 其他部分（含其他 mermaid 块）的正常渲染。
- **FR-011**：渲染后的 Mermaid 图必须只读——不响应点击导航、不改变主画布选中状态、不触发任何对其他实体的跳转（与 spec 005 Related Entities 的"只读"策略一致）。
- **FR-012**：合法的 Mermaid 图必须能正确呈现节点、边、标签、方向（左/右/上/下），并且对状态图的状态名、转移标签、注解（note）等元素做合理排版，不出现节点重叠到不可读或边标签丢失。

#### 侧边栏宽度放宽

- **FR-013**：侧边栏拖拽的宽度上限必须从当前固定的 800px 改为"当前视窗宽度的 2/3"（动态计算），允许用户把侧边栏拖到该上限以内任意宽度。
- **FR-014**：侧边栏拖拽的宽度下限必须保留为一个合理的最小可用值（不大于原下限，避免侧边栏被拖到无法使用），不引入新的更小下限。
- **FR-015**：当视窗尺寸变化（缩放浏览器窗口、改变分屏等）导致已设定的侧边栏宽度超过新的 2/3 上限时，侧边栏实际渲染宽度必须自动 clamp 到新上限，避免侧边栏溢出/遮挡画布。
- **FR-016**：放宽上限不应改变侧边栏的默认初始宽度、不应改变侧边栏的折叠/展开行为、不应改变其他与宽度无关的现有交互。

### 关键实体

- **实体 docstring**：所选 schema 模型类在源码层面写下的文档字符串，作为 About tab 的内容来源；可以是纯文本、Markdown、或包含 mermaid 围栏块的复合文本。
- **渲染后的 Markdown 内容**：docstring 经 Markdown 渲染管线产出的富文本视图，包含段落、列表、表格、代码块，以及由 mermaid 块进一步渲染而成的内嵌图表。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：在 `demo/enterprise_voyager` 中，用户能在双击实体后于侧边栏内打开 "About" tab，并在 1 次点击内看到该实体 docstring 的 Markdown 渲染结果，而无需打开 Source Code 或离开侧边栏。
- **SC-002**：对所有 docstring 中包含 Mermaid 状态/流程图的实体，用户在 About tab 中看到的是**可视化图表**而非源码，可一眼识别节点与转移关系（与读原始 mermaid 源码相比，理解时间显著下降）。
- **SC-003**：在宽度为 1920px 的桌面视窗下，用户能把侧边栏拖宽到至少约 1280px，且在该宽度下阅读宽表格、宽 Mermaid 图、长 Python 行时不再需要横向滚动（或显著减少横向滚动次数）。
- **SC-004**：放宽拖拽上限后，在常见桌面分辨率（1280px–2560px）下侧边栏与主画布布局不出现破损（无重叠、无溢出、无视窗缩放后的"侧边栏比画布还宽"异常）。
- **SC-005**：对所有 docstring 为空或缺失的实体，用户在 About tab 中看到的是一句明确的空状态文案，而不是空白、卡顿或"加载中"假象。

## 假设

- About tab 与 Related Entities 一样，仅在 ER-diagram 模式下显示（其他模式下侧边栏结构不变）。docstring 的概念最贴合 ER-diagram 中的实体类，因此把可见性限定在该模式。
- About tab 在 tab 栏中的位置为**最左**（出现在 Fields 之前）；但侧边栏首次打开时的默认激活 tab 维持 Fields，不因新增 About 而改变（见 Session 2026-07-01 Q1）。
- About tab 的内容来源约定为该 schema 模型类的 Python `__doc__`（类级文档字符串）；当 `__doc__` 为空时直接展示空状态，**不**回退到字段级 `description` 或任何其他字段（见 Session 2026-07-01 Q2）。具体的取值与传输路径（在 schema 元数据里加字段、或新增独立端点）属于实现决策。
- "切换实体保留当前激活 tab" 这一策略对四个 tab（Fields / Source Code / Related Entities / About）一致适用，沿用 spec 005 已建立的模式。
- 侧边栏拖拽下限维持现有的最小可用宽度（约 300px 量级）；该值在本规格中不重新定义具体数字，留给实现沿用现状。
- Mermaid 渲染只读、不接主画布跳转——与 spec 005 中 Related Entities 子图的"只读"决策保持一致。
- Mermaid 的图表主题（明/暗）跟随应用整体主题，不为 About tab 单独引入新的主题切换。
- 不在 About tab 内提供 docstring 的编辑入口、不做 docstring 与源码的双向同步——About tab 是只读视图。
- Mermaid 之外的其他图表语言（PlantUML、Graphviz DOT 等）按普通代码块展示，不在本期支持渲染。
