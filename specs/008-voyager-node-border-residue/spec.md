# 功能规格说明：Voyager ER 图 —— 节点高亮边框残留修复

**功能分支**：`008-voyager-node-border-residue`

**创建日期**：2026-07-03

**状态**：草稿

**输入**：用户描述："现在 voyager 在使用的时候， 如果双击， 会有橙色的效果， 但是如果此时点击了其他 node， 那么原来那个 node 虽然 title 的背景橙色效果没了， 但是边框还会有淡淡的橙色的效果， 看起来有点不舒服，我想解决这个问题。"

## 澄清记录

### Session 2026-07-03

- **Q1**: 这个 bug 在哪些交互路径下出现？Voyager 当前对节点高亮至少有两条路径——(a) 双击节点打开侧边栏（"deep"高亮），(b) 单击节点切换选中（"shallow"高亮、单层邻居）。用户报告的是双击之后"点击其他 node"——这里的"其他 node"是单击还是再次双击？ → **A**: 合理默认——两种路径都覆盖。修复目标是"任何一个被高亮过的节点，在失去焦点后不应残留橙色边框"，不论触发是双击→单击、双击→双击、还是单击→单击。spec 的验收场景按"双击触发后切换到其他节点"为主要场景，但增加边界场景覆盖单击→单击、双击→双击、切换 schema/模式时清空。
- **Q2**: "淡淡的橙色"具体指什么强度？是用户主观感受还是有具体的视觉残留来源？ → **A**: 代码层面的合理默认——当前 `highlightSchemaBanner` 把节点的外框 `polygon[0]` 的 `stroke` 直接设为主题橙色，`clearSchemaBanners` 在清除时只删除了 `data-original-*` 数据属性、未真正把 stroke 还原回原值，所以外框留着上次高亮的橙色 stroke。修复目标是"清除后视觉上完全恢复到未高亮状态"，与从未高亮过的节点像素级一致。
- **Q3**: 修复范围是否包括"切换 schema / 切换 ER-diagram 模式"时的状态清空？ → **A**: 包括。任何让当前高亮失去意义的场景（切换 schema、切换到 voyager 模式、刷新页面、关闭侧边栏）都应顺带清掉残留边框，避免用户回到原 schema 时看到上次未清干净的边框。

## 用户场景与测试 *(必填)*

### 用户故事 1 —— 切换节点后旧节点边框完全清干净（优先级：P1）

开发者或数据探索者正在 Voyager 的 ER 图 tab 中浏览一个 schema。他们双击某个实体 `A` 打开侧边栏——`A` 的标题背景与外框都被高亮为橙色（项目主题色），同时 `A` 的直接邻居与连边也被高亮，便于一眼看清 `A` 在数据模型中的位置。接着他们点击（单击或双击）另一个实体 `B` 想看看 `B` 的关系——此时 `A` 的标题背景橙色消失、`B` 接管高亮，**但 `A` 的外框仍残留一条淡淡的橙色描边**，与画布上其它从未高亮过的节点（无描边或灰色细描边）形成不一致，视觉上有"没清干净"的不适感。用户希望切换节点后，旧节点的视觉状态完全恢复到"从未被高亮过"的样子——外框颜色、宽度、填充与未高亮节点像素级一致。

**为什么是这个优先级**：这是本特性的核心价值——消除一个肉眼可见、且每次节点切换都会触发的视觉瑕疵。Bug 不影响数据正确性或功能可用性，但持续干扰阅读体验，长期用户会反复感知。修复独立交付即可观察、可验证，因此 P1。

**独立测试**：在 `demo/enterprise_voyager` 上打开 ER 图，双击实体 `A`（确认 `A` 标题背景与外框都变橙色），然后单击另一个实体 `B`。截屏对比 `A` 与一个从未被点击过的第三方实体 `C`——验证 `A` 的外框 stroke 颜色、stroke-width、fill 与 `C` 完全一致，无任何残留。

**验收场景**：

1. **给定** ER 图已加载某 schema，**当** 用户双击实体 `A`（高亮生效：标题背景与外框都变橙色），然后单击实体 `B`，**那么** `A` 的外框完全恢复到未高亮状态——stroke 颜色、stroke-width、fill 均与从未高亮过的节点一致，无橙色残留。
2. **给定** ER 图已加载某 schema，**当** 用户双击 `A`，再次双击 `B`，**那么** `A` 同样完全恢复，与单击切换路径行为一致（验收场景 1）。
3. **给定** ER 图已加载某 schema，**当** 用户单击 `A`（shallow 高亮：标题背景与外框变橙、单层邻居高亮），然后单击 `B`，**那么** `A` 外框完全恢复，无橙色残留。
4. **给定** 用户已双击 `A` 触发高亮，**当** 用户切换到另一个 schema（侧边栏重新打开 / 主图重新生成），**那么** 切换后 `A` 不再残留任何橙色描边；新 schema 的所有节点视觉状态干净。
5. **给定** 用户在 ER-diagram 模式下双击 `A` 触发高亮，**当** 用户切换到 voyager 模式（如 use-case 图）再切回 ER-diagram 模式，**那么** 切回后 `A` 不残留任何橙色描边。
6. **给定** 用户双击 `A`，**当** 用户关闭侧边栏（不切换其他节点），**那么** `A` 的高亮是否清除取决于现有项目交互设计——本特性不强制改变关闭侧边栏时是否清高亮的现有行为，但若关闭侧边栏触发了 `clearSchemaBanners` 路径，则该路径同样不得残留橙色描边。
7. **给定** ER 图含自引用关系（如 `Tree.parent_id → tree.id`），**当** 用户双击 `Tree`，**那么** 自环高亮、清除路径同样不残留边框橙色。
8. **给定** 用户快速连续点击多个节点（A → B → C → D），**当** 每次切换发生，**那么** 每个被遗弃的节点都立即清干净，不允许出现"中间态橙色"或"残留累积"。

### 用户故事 2 —— 长会话内多次切换不累积视觉噪声（优先级：P2）

在一次工作会话中，用户可能反复双击 / 单击多个实体来比对它们的字段与关系——每次切换都应让旧节点的视觉状态完全归零，否则多次切换后画布上会累积若干"半褪色"的橙色描边，让人误以为这些节点还在某种"次级选中"状态。修复后，用户在长时间浏览、反复切换几十次后，画布视觉状态与刚加载时一致（除当前真正选中节点外，无任何残留）。

**为什么是这个优先级**：Story 1 修复单次切换；Story 2 验证修复在长会话、多次切换下仍稳定（无回归、无累积），属于质量加固，因此 P2。

**独立测试**：在 ER 图中连续切换 20 次以上节点（混合双击与单击），截屏对比"刚加载完的画布"与"切换 20 次后的画布"（除当前选中节点外），验证无累积性视觉差异。

**验收场景**：

1. **给定** ER 图已加载，**当** 用户连续切换 20 次以上节点（混合双击/单击），**那么** 当前画布上除真正选中节点外，其他节点视觉状态与刚加载时像素级一致（无残留、无累积）。
2. **给定** 用户已多次切换节点，**当** 用户刷新浏览器页面，**那么** 重新加载后的画布无任何橙色描边残留（基线确认）。
3. **给定** 用户在长会话内多次切换 schema、切换显示选项 toggle（cluster display / Hide Reverse Relationships 等）、再切换节点，**当** 上述任一操作触发主图重渲染，**那么** 重渲染后无任何橙色描边残留（高亮状态随重渲染归零）。

### 边界情况

- 节点切换发生在主图重新生成过程中（in-flight）：上一个节点的高亮必须在新图渲染完成前清干净，新图加载完成后只高亮新选中节点。
- 节点点击触发 Related Entities 子图 refetch（spec 007 修复后的行为）：子图 refetch 不影响主图节点的高亮状态——主图高亮由主图交互路径独立管理。
- 自引用关系节点（如 `Tree`）的双击高亮：清除路径必须正确处理自环，不留橙色描边。
- 极快速连续点击（A → B 在 1 帧内）：去抖动后最终状态正确（B 高亮、A 完全清除）。
- 切换 schema 时上一 schema 的高亮状态：必须随 schema 切换归零，不允许跨 schema 残留。
- 切换 ER-diagram 模式 ↔ voyager 模式：模式切换时高亮状态归零。
- 浏览器刷新：刷新后画布无残留（基线一致）。
- 用户在侧边栏打开状态下双击同schema内的另一节点：旧节点清除、新节点高亮，二者均无视觉残留。
- 节点 SVG 元素被 Graphviz 重新生成（如布局算法触发 SVG 替换）：原 SVG 元素上的 stroke 残留不应"复活"到新元素。
- 主题色变更（如果项目支持运行时切换主题）：原橙色高亮的清除逻辑不应硬编码颜色名，应基于"清除时还原到原始属性"而非"涂改回某固定颜色"。

## 需求 *(必填)*

### 功能需求

#### 高亮清除语义

- **FR-001**：当节点的视觉高亮被清除时（无论是切换到其他节点、切换 schema、切换模式、刷新页面、还是关闭侧边栏触发的清除路径），节点的外框 `polygon[0]` 与标题背景 `polygon[1]` 的 `stroke` / `stroke-width` / `fill` 三个 SVG 属性必须**完全恢复到该节点从未被高亮过的状态**——与同画布上从未高亮过的同类节点像素级一致。
- **FR-002**：清除操作必须基于"还原到原始属性"的语义（在节点首次高亮时保存原值、清除时把属性写回原值），**不得**基于"涂改回某个固定颜色"——后者会在主题色、深色模式或自定义渲染场景下失败。
- **FR-003**：清除必须覆盖两条高亮路径——(a) 双击节点打开侧边栏的"deep"高亮、(b) 单击节点切换选中的"shallow"高亮——以及它们之间的任意切换组合（双击→双击、双击→单击、单击→双击、单击→单击）。
- **FR-004**：当主图因任何原因重新生成（切换 schema、切换显示选项 toggle 触发 refetch、模式切换）而替换 SVG 时，新 SVG 上不得继承任何"残留高亮"——即重生成后的画布除当前真正选中节点外，其他节点视觉状态干净。

#### 交互覆盖

- **FR-005**：用户双击实体 A 触发高亮后，单击或双击实体 B 切换选中——A 的视觉高亮必须完全清除（含外框描边），不得残留任何橙色。
- **FR-006**：用户切换 schema、切换 ER-diagram ↔ voyager 模式时，前一状态的所有节点高亮必须随状态转换清零，新画布不继承任何残留。
- **FR-007**：用户连续切换多个节点（混合双击与单击）时，每次切换都让被遗弃的节点立即清干净，不允许出现"中间态橙色"或"残留累积"。
- **FR-008**：用户刷新浏览器页面后，画布除当前真正选中节点外（若刷新前有选中且 URL/state 还原），其他节点视觉状态干净。

#### 不变量

- **FR-009**：本特性不改变现有的"哪个节点应该被高亮"语义——只修复"清除不彻底"的视觉瑕疵，不引入新的高亮规则、不改变触发条件、不调整橙色色值。
- **FR-010**：本特性不影响 Fields / Source Code / Related Entities / About 各 tab 的内容显示，不影响主图渲染管线，不影响其他显示选项 toggle 的行为。
- **FR-011**：本特性不引入新的依赖、不改 SVG 渲染管线的契约形状、不改 Graphviz 输出格式。

### 关键实体

- **节点外框（outer frame）**：Graphviz 输出的 SVG 中每个实体节点的最外层 `<polygon>`（节点边框），是橙色描边残留的主要载体。
- **节点标题背景（title background）**：节点内紧贴外框的第二个 `<polygon>`（标题栏背景），高亮时填充与描边都变橙色，目前清除路径能正确还原。
- **高亮清除**：把之前被改写过的 SVG 属性（`stroke` / `stroke-width` / `fill`）恢复到该节点未被高亮时的原始值的过程。语义是"还原原值"，不是"涂改回固定颜色"。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：在 `demo/enterprise_voyager` 上，用户双击实体 A、然后单击实体 B 后截屏，将 A 与同画布上从未被点击过的第三方实体 C 比较——A 的外框 `stroke` 颜色、`stroke-width`、`fill` 与 C 像素级一致（无橙色残留）。
- **SC-002**：用户在长会话内连续切换 20 次以上节点（混合双击与单击）后，画布上除当前选中节点外，其他节点视觉状态与刚加载时一致——无累积、无半褪色。
- **SC-003**：用户切换 schema、切换 ER-diagram ↔ voyager 模式、刷新浏览器后，新画布无任何橙色描边残留。
- **SC-004**：修复对自引用关系节点（如 `Tree`）、M2M 关系节点、被 Hide Reverse Relationships 模式过滤后仍可见的节点都生效——清除路径覆盖所有当前可见的节点类型，不留特殊角落。
- **SC-005**：修复不改变现有"哪个节点该被高亮"的语义——所有现有交互（双击打开侧边栏、单击切换、邻居高亮、Related Entities 子图跟随）行为与之前一致，仅旧节点的视觉清除更彻底。

## 假设

- 修复目标仅限于"清除后视觉无残留"，不引入新的高亮规则、不改变触发条件、不调整橙色色值（沿用项目主题色 `--q-primary`）。
- "完全恢复到未高亮状态"的判定基准是同画布上从未高亮过的同类节点的 SVG 属性——而非硬编码的固定 RGB 值（避免主题色变化时失效）。
- 修复路径优先考虑"在节点首次高亮时保存 SVG 原始属性、清除时把属性写回"的语义，而非"用固定颜色覆盖"——后者会在主题色变更、深色模式、自定义渲染下失败。
- 修复同时覆盖双击（deep）与单击（shallow）两条高亮路径——不论用户后续用什么方式切换节点，旧节点都得清干净。
- 切换 schema、切换模式、刷新页面、关闭侧边栏等触发清除或重渲染的场景，都应顺带清掉残留——本特性不强制改变这些场景"是否清高亮"的现有决策，但若它们触发了清除路径，清除必须彻底。
- 本特性不引入前端构建工具链、不引入新依赖、不改 Graphviz SVG 输出格式契约。
- 后端无任何改动——本特性纯前端修复（`graph-ui.js` / `graphviz.svg.js` / `store.js` / `vue-main.js` 等可能涉及，具体路径由 plan 阶段评估）。
- 用户浏览器为现代 Chromium 系或 Firefox（与项目其他前端特性一致），不考虑 IE。
- 本特性不引入自动化前端测试（沿用项目"前端无构建工具链、依赖 quickstart 人工验证"惯例），覆盖方式为 quickstart 步骤化人工验证。
