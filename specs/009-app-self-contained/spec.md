# 功能规格说明：业务应用（Application）自包含数据库连接信息

**功能分支**：`009-app-self-contained`

**创建日期**：2026-07-12

**状态**：草稿

**输入**：用户描述："当前代码 mcp 中的 app，是否支持独立导出？比如我有其他几个 nexusx 的项目，然后在一个单独的 nexusx 项目中合并所有的 app……因此 app 需要改进，让 app 自己要包含完整的信息。"

## 用户场景与测试 *(必填)*

### 用户故事 1 —— 跨项目合并多个 app 到统一 mcp 服务（优先级：P1）

开发者在一个组织里维护多个 nexusx 项目——比如 `blog`、`shop`、`auth` 三个独立项目，每个项目有自己的 nexusx 业务模型与数据库。现在希望启动一个"网关型" mcp 服务，把这三个 app 一起暴露给 AI 助手使用。在当前版本下，开发者必须在网关项目里**重新声明**每个 app 的 SQLModel base 类**以及**对应的 session_factory（数据库连接信息）——session_factory 不能跟着 app 一起搬过来，导致"合并"实际上变成"重新布线"。开发者希望：每个子项目自己声明 app 时就把数据库连接信息打包在一起，网关项目只需要 `pip install blog-app shop-app auth-app`、import 进来、丢给 `create_mcp_server(apps=[blog_app, shop_app, auth_app])` 就完成组装——一行 import 一行组装，不再需要"为每个 app 配连接资源"。

**为什么是这个优先级**：这是本特性的核心动机。用户原始问题就是从"合并多个 nexusx 项目"出发的——如果这一步做不到，"自包含"等于没兑现。P1。

**独立测试**：在新建的网关项目中安装三个子项目作为依赖，每个子项目以 Python 包形式暴露自包含的 app 单元；网关项目的全部代码不应超过 10 行即可启动一个能服务三个 app 的 mcp server。验证用 mcp client 调用 `list_apps` 返回三个名字、对每个 app 调 `graphql_query` 都能命中各自数据库。

**验收场景**：

1. **给定** 三个独立 nexusx 子项目（blog/shop/auth）各自发布为 Python 包，**当** 在网关项目 `pip install` 这三个包并在代码中 import 各自暴露的 app 单元、传入 `create_mcp_server(apps=[...])`，**那么** mcp server 启动后能通过 `list_apps` 工具看到全部三个 app，对每个 app 调用 GraphQL 查询能命中各自的数据库，互不串扰。
2. **给定** 子项目里 app 单元的定义包含完整的数据库连接信息（数据库 URL 字符串），**当** 网关项目 import 它时，**那么** 不需要在网关项目里再次提供 session_factory 或数据库 URL——app 自身持有的连接信息足以正常工作。
3. **给定** 三个 app 各自指向不同的数据库实例（不同 URL、不同连接池），**当** mcp server 启动，**那么** 三个 app 的查询彼此完全隔离，任一 app 的数据库故障不影响其他 app 的查询能力。
4. **给定** mcp server 收到正常关闭信号（如进程 SIGTERM、lifespan 退出），**当** server 完成关闭流程，**那么** 三个 app 持有的数据库连接资源被释放，无连接泄漏（可在数据库端观察连接数归零或在日志中确认 dispose 调用）。

---

### 用户故事 2 —— 单个 app 作为可独立使用的 Python 包（优先级：P2）

开发者构建了一个 nexusx 业务应用（例如内部"任务管理"app），希望把它发布给团队其他成员复用。使用者 `pip install task-app` 后，在自己的脚本或服务里直接 import 并使用——既能挂到自己的 mcp server 里组装，也能独立运行查询（用于测试、数据导入脚本、定时任务等）。当前版本下，使用者在 import 后还必须自己造 session_factory 注入才能跑查询，"独立使用"等于半个制品。本特性让 app 自带连接信息后，使用者从安装到首次跑通查询只需要提供数据库 URL（或环境变量）。

**为什么是这个优先级**：这是"独立导出"的字面价值——一个 app 包应该是一个完整可用的单元，而不是半个需要再加工的组件。P1 已经覆盖了"合并"场景；P2 覆盖"单 app 单独使用"，是同一个改造的另一面，优先级仅次于合并。

**独立测试**：在干净的虚拟环境中 `pip install` 一个 app 包，写一段 5 行以内的 Python 脚本：import app、调 GraphQL 查询、打印结果——脚本能成功执行并返回数据。

**验收场景**：

1. **给定** app 包已发布且使用者已 `pip install`，**当** 使用者 import 该 app 并直接调用其 GraphQL 查询能力，**那么** 仅需提供数据库 URL 一个配置项即可成功执行查询返回数据。
2. **给定** 使用者希望对 app 做 schema 内省（查看 SDL、列出可用查询）而不实际连库，**当** 使用者在不提供数据库 URL 的情况下构造 app，**那么** schema 内省与文档查询全部可用；只有真正执行查询时才会因为缺连接而失败。
3. **给定** 使用者已经在自己的代码里维护了一个数据库连接（engine 对象或 session 工厂），**当** 使用者希望让 app 复用自己的连接资源，**那么** app 必须支持接收外部传入的 engine 对象或 session 工厂，且 app 不得在关闭时 dispose 调用方持有的 engine（所有权清晰）。

---

### 用户故事 3 —— 平滑迁移现有 dict 式配置（优先级：P3）

现有 nexusx 用户已经在自己的代码里用字典形式（`{"name": ..., "base": ..., "session_factory": ...}`）声明 app。本特性引入新的自包含 app 单元后，这些用户的代码不应一次性全部失效——应该有一个兼容窗口，让用户在自己的节奏下逐步迁移：旧用法仍然工作但发出弃用警告，给用户足够时间在新版本发布前更新代码；新版本发布一段时间后再彻底移除旧用法。

**为什么是这个优先级**：用户体验问题，不阻塞核心功能。但如果没有这个兼容窗口，升级路径会非常痛，影响用户对 nexusx 的信任。P3。

**独立测试**：用现有 nexusx 仓库自带的全部 demo / 测试 / 技能模板（skills/nexusx-4phase）跑一遍——它们都使用 dict 形式——在迁移前的版本上应能继续运行（带弃用警告）；迁移完成、移除 dict 形式后，所有示例和测试都已切到新形式。

**验收场景**：

1. **给定** 用户代码使用字典形式声明 app，**当** 用户在新版本 nexusx 上运行该代码，**那么** 程序仍能正常运行并完成所有原有功能；同时输出一条 `DeprecationWarning` 明确告知该用法已弃用、未来版本将移除、并给出新形式的迁移示例。
2. **给定** 用户已经把所有 app 声明迁移到新形式，**当** 用户在新版本上运行代码，**那么** 不再有任何弃用警告；功能与旧版本完全等价。
3. **给定** 字典形式声明里同时提供了多个互斥的连接信息字段（如同时给数据库 URL 与 session 工厂），**当** 系统校验该配置，**那么** 立即报错并明确指出"只能提供其中之一"，不进入不可预测的状态。

### 边界情况

- **零连接信息场景**：开发者只想做 schema 内省、生成 SDL 文档或运行单元测试（无真实数据库），app 必须能在不提供任何连接信息的情况下构造——只有真正执行 GraphQL 查询时才因缺连接失败。
- **资源所有权判定**：app 通过 URL 字符串自己造的连接资源，所有权归 app，关闭时由 app 释放；app 通过外部传入的 engine 对象或 session 工厂接收的资源，所有权归调用方，app 关闭时不释放。三选一的判定必须明确且无歧义。
- **多次关闭幂等性**：app 关闭操作必须幂等——重复调用、或在 lifespan 已经清理过后再次调用，不应抛异常或造成副作用。
- **共享 engine 的多 app 场景**：多个 app 复用同一个外部 engine 对象时，关闭任一 app 不应 dispose 这个共享 engine，engine 的生命周期完全由创建者管理。
- **重复名称与别名冲突**：多个 app 的名称、别名之间发生冲突时，必须在构造期立即报错并明确指出冲突项；不能让冲突潜伏到运行期。
- **现有简单 mcp 服务（单 app）与 UseCase mcp 服务**：本次改造只覆盖多 app 场景；单 app 与 UseCase 这两条平行路径不在本次范围内，沿用现状。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统 MUST 提供"业务应用"（Application）这一抽象作为 nexusx 中可独立导出、可合并的最小单元，封装业务模型（SQLModel base）+ 数据库连接信息 + 元数据（名称、描述、查询/变更描述、别名）。
- **FR-002**：业务应用 MUST 支持三种连接信息接收方式，三选一：(a) 数据库 URL 字符串（app 自己构造并拥有连接资源）；(b) 已构造的异步引擎对象（app 包装但不拥有）；(c) 会话工厂（app 包装但不拥有，兼容旧用法）。
- **FR-003**：业务应用 MUST 支持零连接信息构造（schema-only 模式）——此时 schema 内省、SDL 生成、文档查询全部可用，仅 GraphQL 查询执行因缺连接失败。
- **FR-004**：系统 MUST 提供一个 `create_mcp_server(apps=[...])` 入口，接受一个或多个业务应用作为输入，启动一个对外暴露这些应用的 mcp 服务。
- **FR-005**：多个业务应用 MUST 能在同一个 mcp 服务中并存，每个应用拥有独立的数据库连接与独立的 GraphQL schema，互不串扰。
- **FR-006**：系统 MUST 在 mcp 服务生命周期结束时（lifespan 退出）正确释放每个应用自身拥有的数据库连接资源；非自身拥有的资源（外部传入的 engine / session 工厂）不得被释放。
- **FR-007**：关闭操作 MUST 幂等——重复调用或异常路径下重复触发不应抛错或产生副作用。
- **FR-008**：系统 MUST 在构造期检测并拒绝以下错误配置，抛出明确错误：重复的应用名称、应用名称与别名的冲突、别名之间的冲突、应用别名与自己的名称相同、同时提供多种互斥的连接信息。
- **FR-009**：系统 MUST 接受旧式字典配置作为兼容输入；字典输入 MUST 在使用时触发 `DeprecationWarning`，警告内容包含弃用时间窗口提示与新形式迁移示例。
- **FR-010**：业务应用 MUST 支持以 Python 包形式发布——子项目 `pip install` 后，使用方可通过 `import` 直接拿到一个可用的 app 单元，无需再为该 app 配置连接资源。
- **FR-011**：业务应用 MUST 可作为异步上下文管理器使用（`async with Application(...) as app: ...`），离开上下文时自动释放自身拥有的连接资源。
- **FR-012**：系统 MUST 通过项目公共 API（如 `nexusx.mcp` 模块的导出列表）暴露 `Application` 这一新抽象，让使用方无需深入子模块即可使用。
- **FR-013**：当业务应用通过 URL 字符串接收连接信息时，系统 MUST 在所有面向用户的输出中（错误消息、异常字符串、日志、调试信息）**自动脱敏 URL 中的密码字段**——把密码部分替换为掩码（如 `postgresql://user:***@host/db`）；完整 URL 仅保留在内存中用于实际建立连接，不得出现在任何可被日志采集/异常追踪系统捕获的文本里。

### 关键实体

- **业务应用（Application）**：nexusx 中一个独立的业务模块，封装 SQLModel 业务模型、数据库连接信息（URL / engine / session 工厂任选其一）与 GraphQL 元数据（名称、描述、别名）。是可独立导出、可合并到多 app mcp 服务的最小单元。
- **多应用 mcp 服务（Multi-App MCP Server）**：把多个业务应用组装到一个对外接口（mcp server）中，通过 `app_name` 参数路由工具调用到对应的应用。
- **资源所有权（Resource Ownership）**：业务应用对数据库连接资源的持有关系——通过 URL 自造的资源归 app 所有（关闭时释放），外部传入的资源归调用方所有（app 不释放）。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**：开发者在网关项目中组装来自 3 个不同子项目的业务应用到统一 mcp 服务，从 `pip install` 到 mcp server 启动成功的全部组装代码不超过 10 行。
- **SC-002**：业务应用作为 Python 包发布后，使用者从安装到首次成功执行 GraphQL 查询返回数据，需要配置的项不超过 1 个（数据库 URL）；从安装到完成 schema 内省查询（不需要连库）零配置。
- **SC-003**：现有 nexusx 用户从旧版本升级到新版本时，所有现有 demo、测试、技能模板（nexusx-4phase）在弃用警告下仍可运行——升级不阻塞业务；用户在自己的代码迁移到新形式后，弃用警告消失，功能与旧版本等价。
- **SC-004**：mcp 服务在正常关闭流程（SIGTERM、Ctrl+C、lifespan 退出）后，所有应用自身持有的数据库连接资源被释放——可在数据库端通过连接数监控归零、或在应用日志中确认 dispose 调用记录来验证；多次关闭或异常路径下重复释放不产生副作用。
- **SC-005**：3 个独立子项目的 app 合并到网关项目后，每个 app 各自独立的 GraphQL schema、entity 名称空间、数据库连接——任一 app 的 schema 变更、查询失败、数据库故障都不影响其他两个 app 的正常运行（隔离性）。

## 假设

- 使用者使用 SQLAlchemy/SQLModel 兼容的异步数据库驱动（SQLite + aiosqlite、PostgreSQL + asyncpg 等），通过数据库 URL 字符串描述连接。
- 使用者熟悉 Python 包管理与虚拟环境工作流；"独立导出"以 Python 包（pip 安装）作为交付形态，不涵盖二进制制品或跨语言联邦。
- 现有 dict 式配置用户在弃用警告发出的兼容期内逐步迁移——兼容期为一个 minor 发布周期，之后移除。
- 单 app 简化路径（`create_simple_mcp_server`）与 UseCase mcp 路径（`create_use_case_graphql_mcp_server`）本次不在改造范围，沿用现状；作为后续工作推进。
- 多 app 场景下，每个 app 默认持有独立的数据库连接池；如需共享连接池，使用方通过传入 engine 对象的方式自行管理。
- 业务应用在 mcp 服务外被独立使用（脚本、定时任务等）时，使用方负责自己管理 app 的生命周期（`async with` 或手动关闭）。
- "业务应用包"以 Python 包（wheel / sdist）形式发布到 PyPI 或私有索引；联邦场景下的多进程 / 多语言集成不在本特性范围。

## 澄清记录

### Session 2026-07-12

本特性的核心方向通过 AskUserQuestion 与用户对齐：

- **Q1**：「app 自包含完整信息」要做到哪个深度？ → **A**：深度 A——app 持数据库 URL 自己构造 engine 与 session 工厂，子项目 `pip install` 后 import 即用，不需要合并项目提供连接资源。代价：合并项目无法统一连接池/事件钩子；需要新增生命周期 API。用户接受代价，因为合并场景下每个 app 自带独立数据库是常态。
- **Q2**：旧的 AppConfig 字典形式要不要升级为类？ → **A**：升级为类，但类名改为 `Application`（不叫 `GraphQLApp`）。旧的 `AppConfig` TypedDict 作为弃用兼容窗口的输入形式保留一段时间。

实施前的设计探索发现一个关键约束：

- **Q3**：业务应用是否必须强制要求"恰好提供一个"连接信息字段（URL/engine/session factory 三选一）？ → **A**：否——必须支持零连接信息构造（schema-only 模式）。现有 nexusx 测试套件中大量 dict 配置只提供 `name`/`base`/`description`（用于 SDL 内省测试），强制要求连接信息会一次性打破全部测试；同时 schema-only 模式对文档生成、Schema 浏览等场景有独立价值。因此正确规则是"至多提供一个"，零也合法。
- **Q4**：数据库 URL 含密码时，nexusx 在日志/错误消息中如何处理？ → **A**：自动脱敏——nexusx 在所有面向用户的输出（错误消息、异常字符串、日志、调试信息）中默认把 URL 中的密码字段替换为掩码 `***`；完整 URL 仅保留在内存中用于实际建立连接。理由：nexusx 作为 schema-to-API 工具集，用户体量上来后日志聚合/异常上报场景下凭据泄漏是真实安全风险；脱敏实现成本低（SQLAlchemy 自带 `URL.render_as_string(hide_password=True)`），而"调用方自己负责"等于不设防。详见 FR-013。
